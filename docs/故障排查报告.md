## 一、系统架构概述

```
用户上传 PDF 发票 + Excel 账单
  → Stage 1: quick_extract_totals（快速提取每个 PDF 的总金额）
    → 规则抽取（文本解析）
    → 缓存检查
    → AI 抽取（MiMo API via Anthropic Messages API）
  → Stage 1 比对: compare_by_warehouse（仓库级总金额比对）
  → 如果总金额不一致 → Stage 2: extract_invoice_items（员工明细抽取）
    → 规则抽取
    → AI 文本抽取
    → AI 图片抽取（PDF 渲染为图片 → MiMo API）
  → compare_labor_items（员工级比对）
  → 质量评估 → 如果 warning/critical → 重试抽取
  → 生成差异报告
```

**AI 服务**: MiMo token plan (https://token-plan-cn.xiaomimimo.com/anthropic)  
**模型**: mimo-v2.5  
**API 格式**: Anthropic Messages API (`/v1/messages`)

## 二、故障时间线

### 故障 1: 服务器阻塞，API 超时

**用户提报**: 批次 `labor_20260531_213529_193348_0f89a714` 一直读取不出结果

**现象**: 
- 批次状态卡在"抽取中"
- 所有 API 请求超时（curl 无法连接）
- 前端轮询拿不到结果

**日志**:
```
无后续日志，服务器完全无响应
```

**根因分析**:
- `_run_labor_extract_compare` 是同步函数，通过 FastAPI `BackgroundTasks` 执行
- FastAPI 对同步 BackgroundTasks 直接在 uvicorn 主线程执行
- 整个抽取链路（AI HTTP 调用、PDF 渲染、线程池）阻塞了主线程
- 其他所有请求（包括前端轮询 GET）被阻塞

**修复措施** (`d341996`):
```python
# 之前
background_tasks.add_task(_run_labor_extract_compare, run_id)

# 之后
asyncio.get_event_loop().run_in_executor(None, _run_labor_extract_compare, run_id)
```

**结果**: 服务器不再阻塞，API 可正常响应

### 故障 2: Google Fonts 国内被墙

**用户提报**: 页面进不去

**现象**: 页面白屏/加载卡住

**根因**: HTML 中导入了 `fonts.googleapis.com`，国内无法访问，浏览器卡在字体加载

**修复措施** (`3345747`):

- 改用系统字体回退：`'DM Sans', -apple-system, BlinkMacSystemFont, 'SF Pro Display'`

**结果**: 页面可正常加载

### 故障 3: 并发 AI 请求导致卡死（第一次）

**用户提报**: 批次 `labor_20260531_220855_952752_64744ba4` 卡住

**现象**:
- 批次状态: "抽取中"，阶段: "Stage 1: 快速抽取总金额"
- 日志显示 "并发数=6"，7 个 PDF 同时发起 AI 请求
- 无任何 "PDF X/7 完成" 日志

**日志**:
```
22:26:12 [extract] INFO: 开始快速总金额抽取: 7 个 PDF
22:26:12 [extract] INFO: 快速总金额抽取: 7 个 PDF, 并发数=6
（无后续日志）
```

**验证**:
- 单个 PDF 调 MiMo: 5.3 秒 ✅
- 7 个 PDF 并发调 MiMo: 超时卡死 ❌

**根因**: MiMo 服务对并发图片请求有限流/连接池限制，6 个并发导致全部超时

**修复措施** (`ab7b325` + `79adbc4`):

```python
# config.py

"parallel_image_render_workers": _env_int("PARALLEL_IMAGE_RENDER_WORKERS", 2),  # 4→2
```

**注意**: 第一次只改了 extract.py 的 `get` 默认值，但 config.py 已经配了 6，所以没生效。必须改 config.py 源头。

**结果**: 并发数降到 2，Stage 1 可正常完成

### 故障 4: AI 图片抽取返回 0 条（evidence 过滤）

**用户提报**: 批次 `labor_20260531_223531_482559_400f68d5` 失败

**现象**:
- Stage 1 ✅ 完成（7 个 PDF 都成功提取总金额）
- Stage 2 ❌ 失败: "AI 抽取失败：AI 图片抽取返回 0 条员工明细"

**日志**:
```
22:36:17 [app] INFO: [C/D] 开始抽取员工明细: 4 个 PDF, 10 行 Excel
22:36:17 [extract] INFO: 规则抽取失败(0条)，进入 AI 抽取流程
22:36:17 [app] ERROR: 抽取失败(ValueError): AI 抽取失败：AI 图片抽取返回 0 条员工明细
```

**根因分析**:
- AI 确实返回了数据（缓存中有 27 条记录）
- 但 `_looks_like_employee_row` 函数过滤太严格：
  ```python
  if amount > 0 and not any(marker in evidence for marker in ("$", "amount", "bill", "total", "charge", "invoice")):
      return False
  ```
- 图片 PDF 的 AI 抽取返回的 `evidence_text` 常被截断，如 `"Betti ... 12.14"`
- `"... 12.14"` 不含 `$` 等标记词 → 21 条全被过滤

**验证**:
```
p2 缓存: 21 条 → normalize 后: 0 条（全部被 evidence 过滤）
```

**修复措施** (`bacd855`):
```python
# 移除 evidence 标记检查，只保留 amount>0 和合理人名检查
def _looks_like_employee_row(employee_name: str, row: Dict[str, Any]) -> bool:
    amount = parse_number(row.get("amount"))
    if amount == 0:
        return False
    letters = re.findall(r"[A-Za-zÀ-ÖØ-öø-ÿ]", employee_name)
    if len(letters) < 3:
        return False
    return True
```

**结果**: p2 缓存从 0 条恢复到 21 条

### 故障 5: AI 返回 thinking block，无实际结果

**用户提报**: 批次 `labor_20260531_225901_067134_5d0900ff` 卡住

**现象**:
- Stage 1 卡住，无 "PDF X/7 完成" 日志
- AI 调用有响应但返回空

**日志**:
```
23:09:27 [extract] INFO: 快速总金额抽取: 7 个 PDF, 并发数=2
（无后续日志）
```

**验证**:
```python
# 直接调 MiMo API
response = {
    "stop_reason": "max_tokens",
    "usage": {"input_tokens": 50, "output_tokens": 1024},
    "content": [{"type": "thinking", "thinking": "The user wants..."}]  # 只有 thinking，没有 text
}
```

**根因**:
- MiMo API 默认启用 thinking（推理模式）
- `max_tokens=1024` 太小，所有 token 消耗在 thinking 上
- 没有剩余 token 输出实际结果
- `_post_anthropic_completion` 只提取 `type: "text"` 的 block → 返回空

**关键代码**:
```python
# _post_anthropic_completion 只提取 text
for block in data.get("content", []):
    if block.get("type") == "text":  # thinking block 被忽略
        content += block["text"]
```

**修复措施** (`a52e848`):
```python
# 1. 所有直接调用 Anthropic API 的地方加 thinking=disabled
payload = {
    "model": model,
    "max_tokens": 4096,  # 1024→4096
    "thinking": {"type": "disabled"},  # 禁用 thinking
    ...
}

# 2. _post_anthropic_completion 传递 thinking 参数
if "thinking" in payload:
    anthropic_payload["thinking"] = payload["thinking"]
```

**涉及函数**:
- `_extract_total_with_ai_image`（Stage 1 图片抽取）
- `_extract_total_anthropic`（Stage 1 文本抽取）
- `_post_anthropic_completion`（通用 Anthropic API 调用）

**结果**: AI 正确返回 text block，包含实际数据

### 故障 6: 重试机制失败导致整体报错

**用户提报**: 批次 `labor_20260531_230907_169036_f77c8257` 失败

**现象**:
- Stage 1 ✅ 完成
- Stage 2 第一次抽取 ✅ 7 条记录
- 比对完成: 质量=warning，4 条问题
- 触发重试 → 重试失败 → 整体报错 "AI 图片抽取返回 0 条"

**日志**:
```
23:09:47 [extract] INFO: 规则抽取失败(0条)，进入 AI 抽取流程
23:09:47 [extract] INFO: [D] 发起 Anthropic API 请求 (via _post_chat_completion)
23:09:47 [extract] INFO: [E] Anthropic 响应成功: stop_reason=end_turn, output_tokens=372
23:09:47 [extract] INFO: [D] 发起 Anthropic API 请求 (via _post_chat_completion)
23:09:47 [extract] INFO: [E] Anthropic 响应成功: stop_reason=end_turn, output_tokens=364
23:09:47 [app] INFO: [C/D] 员工明细抽取完成: 7 条记录
23:09:47 [app] INFO: [G] 比对完成: 质量=warning, 问题=4条
23:09:47 [app] INFO: 质量为 warning，尝试重试...
23:09:47 [extract] INFO: 规则抽取失败(0条)，进入 AI 抽取流程
23:09:47 [app] ERROR: 抽取失败(ValueError): AI 抽取失败：AI 图片抽取返回 0 条员工明细
```

**根因**:
- 第一次抽取成功（7 条），但质量=warning 触发重试
- 重试时 `_post_chat_completion` 调用了 MiMo API（有响应）
- 但 `_extract_with_ai_images` 返回 0 条（原因待查，可能是 `image_pages` 为空或 `_normalize_ai_rows` 过滤）
- 重试失败后直接抛异常，**丢弃了第一次成功的结果**

**修复措施** (`302212a`):
```python
# _retry_if_better: 重试异常或返回0条时，保留原始结果
try:
    retry_pdf_rows = extract_invoice_items(...)
except Exception as exc:
    logger.error(f"重试抽取异常，保留原始结果: {exc}")
    extraction_quality["retryAttempted"] = True
    extraction_quality["retryApplied"] = False
    return pdf_rows, comparison, extraction_quality  # 保留原始结果

if not retry_pdf_rows:
    logger.warning("重试抽取返回 0 条，保留原始结果")
    extraction_quality["retryAttempted"] = True
    extraction_quality["retryApplied"] = False
    return pdf_rows, comparison, extraction_quality  # 保留原始结果
```

**结果**: 重试失败不再报错，保留第一次成功的结果

## 四、未解决的问题

### 4.1 并发=2 仍然卡死（最高优先级）

**问题**: MiMo 服务对并发图片请求处理不稳定

**建议修复方向**:
1. 并发数降到 1（串行），牺牲速度换稳定性
2. 改用异步 HTTP 客户端（httpx/aiohttp），避免线程阻塞
3. 添加单个请求的超时断开（当前 180s 太长）
4. 添加 MiMo 服务健康检查，失败时快速降级

### 4.2 重试机制的根因未查清

**问题**: 重试时 `_extract_with_ai_images` 返回 0 条，但模拟测试正常

**可能原因**:
- `_render_pdf_pages_to_images` 在重试时返回空（文件被占用？）
- `_apply_image_page_policy` 过滤掉了所有图片
- `_normalize_ai_rows` 过滤掉了所有行
- 需要在 `_extract_with_ai_images` 入口添加 `image_pages` 长度日志

### 4.3 AI 返回不稳定

**问题**: 同一个 PDF，有时返回带 `$` 的 evidence，有时返回不带的

**建议**:
- 在 prompt 中明确要求 evidence 包含金额
- 或放宽 `_looks_like_employee_row` 的过滤条件（已部分修复）
