Vibecoding Case Study

招聘奖金平台 · 从 Excel 到工作台

下载原始模板 查看当前平台

我怎么和 AI 一起，把一张招聘奖金 Excel 模板做成了一个可用的平台。

这个页面不是技术文档，而是一个从 0 到 1 的过程演示。它把最开始的 Excel 模板、我给 AI 的 prompt、我用到的 skills、平台早期示意图，以及现在的平台工作台都放在一页里，方便别人理解什么叫做“人负责业务判断，AI 负责把想法快速变成产品”。

起点

Excel 模板 先把招聘奖金和内推奖金的表格逻辑摸清，再交给代码稳定复现。

中间过程

Prompt + Skills 我不断告诉 AI 业务范围、流程边界和页面形态，AI 负责落地。

现在

月度核算工作台 支持批次、待确认、最终结果、差异报告和归档。

核心认知

不是魔法 不是 AI 自动长出系统，而是人和 AI 持续协作、逐步收敛。

01 · 从 0 到 1 的推进路径

这不是“一次生成”，而是不断迭代

我先用 Excel 验证业务规则，再用 prompt 把需求讲给 AI，之后让 AI 帮我搭出平台雏形、收敛流程、补测试、改前端，最后才得到现在这个更像产品的版本。

1

先有表

最开始的核心资产不是代码，而是一个能算出结果的 Excel 模板。

2

讲给 AI

通过 prompt 把“要解决什么问题、要保留哪些流程”说清楚。

3

搭雏形

先做能跑的版本，不追求一步到位，先把上传、计算、导出打通。

4

收敛流程

逐步加入批次留痕、待确认、最终汇总、线下差异复核等真实工作流。

5

产品化

让它不只是脚本，而是给业务同事也能用的月度核算工作台。

03 · 代表性 Prompt

我不是写代码，我是在不断给 AI 下业务指令

下面这些不是逐字聊天记录，而是根据项目现状还原出来的“代表性 prompt”。它们体现了 vibecoding 的本质：业务方负责表达意图，AI 负责把意图翻译成代码和界面。

Prompt 1 · 定边界

先把 Excel 产品化

把这套招聘奖金 Excel 模板做成一个小平台。 先不要用 AI 来算金额，金额计算必须是确定性的 Python 规则引擎。 平台先支持上传月度数据、读取规则库、输出招聘奖金和内推奖金结果。

这类 prompt 的作用是让 AI 明白：先做“可控、可验证”的产品，而不是追求玄学自动化。

Prompt 2 · 收敛流程

把真实月度工作流加进去

平台不要只做单次上传页。 我要的是一个月度核算工作台：上传导入表后生成批次，保留初算结果、待确认表、最终结果和差异报告。 如果有人工判断节点，就让平台支持下载待确认表再回传确认。

这一步是从“脚本”走向“平台”的分水岭，因为它开始贴近真实业务动作。

Prompt 3 · 做成别人能看懂的页面

前端不只是能用，还要能讲故事

帮我把页面做成更像工作台，而不是开发者测试页。 要能看到流程步骤、批次、指标、筛选器和统一明细表。 我想让非技术同事也能看明白这个平台在做什么。

这类 prompt 解决的是“表达层”的问题，也就是把功能变成更容易传播和演示的形态。

04 · 我运用的 Skills

Skill 不是噱头，而是 AI 的工作方法

对非技术同学来说，可以把 skill 理解成“给 AI 的工作 SOP”。不同 skill 负责不同阶段：先想清楚，再规划，再做页面，再验证结果。

Process

`using-superpowers`

让 AI 先检查当前任务应该走什么方法，而不是一上来就乱写代码。

Design

`brainstorming`

先把要做的页面结构、内容边界、表达方式说清楚，再开始落地。

Planning

`writing-plans`

把想法拆成可执行步骤，例如先锁边界、再改前端、再收 API、最后验证。

UI

`frontend-design`

把“能用的页面”做成“有表达力的页面”，适合对外演示和给小白讲清楚。

Quality

`verification-before-completion`

不是写完就算完，而是要求 AI 用测试和页面检查证明它真的可用。

06 · 做到现在意味着什么

这个项目已经从“会算”走到了“能协作”

真正能给别人启发的，不是“AI 一下子做出了平台”，而是你怎么把复杂业务拆开、讲清、验证、再持续交给 AI 迭代。

业务层

把隐性经验显性化

  • Excel 模板被整理成清晰的导入入口。
  • 规则库被拆成结构化 sheet，方便维护。
  • 待确认节点被从脑内判断变成标准流程。

产品层

把单次计算变成月度工作台

  • 支持批次留痕和历史查看。
  • 支持初算、确认、最终结果、差异报告。
  • 支持统一明细表和筛选器，方便复核。

学习层

给非技术同学的启发

  • 先从你最熟的业务表格开始，不要先想技术栈。
  • prompt 的本质是把业务边界讲清楚。
  • AI 最擅长的是持续协作，不是一次神奇生成。

07 · 怎么推广给其他伙伴用

从本地跑起来，到网页部署，再到软件封装

这个平台如果要给更多伙伴用，不一定一上来就做成大而全系统。更实际的路线是先让本地可运行，再做局域网或服务器网页，最后再根据使用人数决定要不要封装成桌面软件。

阶段 1

先在本地电脑运行

适合你自己先用，或者一两个熟悉流程的伙伴试跑。重点不是“部署”，而是让流程稳定，确保 Excel 上传、初算、待确认、最终结果都走得通。

python3 -m uvicorn bonus_platform.app:app --reload --port 8000 打开浏览器： http://127.0.0.1:8000 演示页： http://127.0.0.1:8000/vibecoding-story.html
  • 优点：最快，改完代码立刻能验证。
  • 适合：产品还在迭代期，流程还没有完全定型。
  • 要求：使用者电脑有 Python 环境，能运行依赖。

阶段 2

部署成内部网页

当流程基本稳定后，更推荐放到一台内部服务器上，让大家通过浏览器访问。这样你不用每个人都教一遍环境安装，也方便统一更新规则库和代码版本。

  • 思路：把 FastAPI 服务部署到公司内网服务器或云主机。
  • 入口：给同事一个固定 URL，大家直接网页登录使用。
  • 配套：规则库、导入模板、批次文件都集中存放，避免各自电脑版本不一致。
  • 下一步：可以补账号权限、操作日志、共享文件目录。

阶段 3

封装成桌面软件

如果未来希望 HR 或薪酬同事双击就能打开，不关心命令行，也可以把这个网页平台再包成桌面应用。底层还是同一套逻辑，只是外面套一层更友好的壳。

  • 思路：把现有 Web 页面 + 后端服务封装成桌面客户端。
  • 体验：用户看到的是“软件”，实际用的还是现在这套平台流程。
  • 适合：伙伴越来越多，但又不想正式走完整 IT 系统接入流程。
  • 注意：桌面封装解决的是安装体验，不会替代权限、备份、数据治理这些事情。

实际推广时，最稳妥的顺序通常是：先自己本地跑顺，再给少数伙伴试用，再做内部网页版本，最后再决定是否需要封装成软件或接入更正式的企业系统。
